# Platform definitions

Canonical terms for Rhesis testing. Agents and humans use the same vocabulary.

<Callout type="info">
  **For AI agents:** fetch clean markdown at `https://docs.rhesis.ai/docs/agent-skill/definitions.md`. Full machine index: `https://docs.rhesis.ai/llms.txt`.
</Callout>

## Core entities

| Term | Definition | Not the same as |
| --- | --- | --- |
| **Endpoint** | The AI application under test — URL, request/response mapping, registered in a project. | Behavior, test set, MCP server |
| **Behavior** | One observable expectation the endpoint should meet. Tags each **test** at generation time. | Category, topic, metric |
| **Metric** | How a test is **scored** — evaluation prompt, threshold, and **`metric_scope`**. Linked to behaviors. | Behavior |
| **Test set** | Named collection of **tests** with one `test_type` for all tests in the set. | Project, behavior list |
| **Test** | One case inside a test set — a **prompt** (Single-Turn) or a **goal** (Multi-Turn). | Test run, test result |
| **Test run** | One execution of a test set against an endpoint. | Test set |
| **Test result** | Outcome for one test in one run — status, metric scores. | Metric definition |
| **Source** | Knowledge document for grounding **Single-Turn** test generation. | Tool connection, chat paste |
| **Tag** | Org label on behaviors/metrics via `assign_tag`. | Category, topic |

## Test shape

| Term | Definition |
| --- | --- |
| **Single-Turn** | One user prompt → one assistant response. `test_type` on the test set. |
| **Multi-Turn** | A conversation goal; Penelope drives multiple turns. |
| **`metric_scope`** | List on each metric: `["Single-Turn"]`, `["Multi-Turn"]`, or both. Platform **drops** metrics whose scope does not match the test's type. |
| **`score_type`** | `numeric` or `categorical` — how the evaluator returns a score. |

## Common confusions

**Behavior vs metric** — Behavior = *what* should happen. Metric = *how* you measure it.

**Single-Turn test vs Single-Turn metric scope** — A Multi-Turn test set only runs metrics with `"Multi-Turn"` in `metric_scope`. Plan both explicitly.

**PRD section title vs behavior** — "Security" is not a behavior. Split numbered requirements into testable expectations.
