# Adaptive Testing

Adaptive Testing is a topic-based workflow for expanding and maintaining single-turn test sets over time.
Instead of treating a test set as static, you can organize tests into topic trees, generate outputs in bulk,
evaluate results with a selected metric, and iterate with AI-generated suggestions.

## When to Use Adaptive Testing

Use Adaptive Testing when you need to:

- Grow coverage across specific risk areas or product domains
- Re-evaluate existing tests without regenerating all outputs
- Curate suggestions before saving them to your test set
- Keep one evolving test set instead of creating many one-off sets

## End-to-End Workflow

Adaptive Testing in the UI follows this loop:

1. Create an adaptive test set
2. Organize tests in topics
3. Generate outputs from a selected endpoint
4. Evaluate tests with a selected metric
5. Generate and review suggestions
6. Accept selected suggestions into the test set

<Callout type="info">
  Topic operations are hierarchical. Renaming a topic cascades to child topics and tests; removing a topic
  removes subtopics and moves tests to the parent topic.
</Callout>

## Adaptive Settings (v0.6.12+)

Adaptive testing settings let you store defaults on the adaptive test set itself:

- **Default endpoint** used when `endpoint_id` is not provided
- **Default metric list** used when `metric_names` is not provided

The UI **Edit settings** dialog writes these defaults once, then later runs can use them without reselecting values each time.

| Field | Type | Required | Notes |
| --- | --- | --- | --- |
| `default_endpoint_id` | `uuid \| null` | No | If provided, must belong to your org and be accessible |
| `metric_ids` | `uuid[] \| null` | No | Replaces the assigned metric list atomically |

<CodeBlock filename="adaptive-settings.sh" language="bash">
{`# Read current adaptive settings
curl -X GET "$RHESIS_BASE_URL/adaptive_testing/$TEST_SET_ID/settings" \\
  -H "Authorization: Bearer $RHESIS_API_TOKEN"

# Update default endpoint + metric list
curl -X PUT "$RHESIS_BASE_URL/adaptive_testing/$TEST_SET_ID/settings" \\
  -H "Authorization: Bearer $RHESIS_API_TOKEN" \\
  -H "Content-Type: application/json" \\
  -d '{
    "default_endpoint_id": "00000000-0000-0000-0000-000000000000",
    "metric_ids": ["11111111-1111-1111-1111-111111111111"]
  }'`}
</CodeBlock>

<Callout type="info">
  When settings are present, output generation and evaluation requests can omit
  endpoint and metric parameters and still run with the configured defaults.
</Callout>

## Generate Outputs and Evaluate with Overwrite Control

Two actions drive most iteration cycles:

- **Generate outputs**: invoke an endpoint for test inputs and store output in test metadata
- **Evaluate**: run a selected metric against test input/output pairs and store label and score in test metadata

Both actions support an `overwrite` option.

| Parameter | Type | Default | Behavior |
| --- | --- | --- | --- |
| `topic` | `string \| null` | `null` | Limits processing to a topic |
| `include_subtopics` | `boolean` | `true` | Includes descendant topics when `topic` is set |
| `overwrite` | `boolean` | `false` | Replaces existing outputs/results instead of skipping |
| `test_ids` | `string[] \| null` | `null` | Optional explicit subset of tests |

<Callout type="info">
  With `overwrite=false`, tests that already have outputs or evaluation labels are skipped. The response includes
  `generated` or `evaluated`, plus `skipped` and `failed` counts.
</Callout>

<Callout type="info">
  In the Adaptive Testing UI, output generation and evaluation currently run with overwrite enabled by default.
  Use the API directly if you need explicit skip behavior with `overwrite=false`.
</Callout>

### Generate Outputs API Example

<CodeBlock filename="generate_outputs.sh" language="bash">
{`curl -X POST "$RHESIS_BASE_URL/adaptive_testing/$TEST_SET_ID/generate_outputs" \\
  -H "Authorization: Bearer $RHESIS_API_TOKEN" \\
  -H "Content-Type: application/json" \\
  -d '{
    "endpoint_id": "00000000-0000-0000-0000-000000000000",
    "topic": "Safety/Jailbreak",
    "include_subtopics": true,
    "overwrite": false
  }'`}

## Suggestions Workflow

Suggestion endpoints are non-persisted until you accept results in the UI.

1. `POST /adaptive_testing/{id}/generate_suggestions`
2. `POST /adaptive_testing/{id}/generate_suggestion_outputs`
3. `POST /adaptive_testing/{id}/evaluate_suggestions`
4. Accept selected suggestions to create real tests in the set

| Suggestion parameter | Type | Default | Notes |
| --- | --- | --- | --- |
| `num_examples` | `int` | `10` | Existing tests sampled as examples |
| `num_suggestions` | `int` | `20` | Requested number of suggestions |
| `topic` | `string \| null` | `null` | Optional topic focus |
| `user_feedback` | `string \| null` | `null` | Optional guidance text to steer suggestion generation (max length `1000`) |

<CodeBlock filename="generate_suggestions.sh" language="bash">
{`curl -X POST "$RHESIS_BASE_URL/adaptive_testing/$TEST_SET_ID/generate_suggestions" \\
  -H "Authorization: Bearer $RHESIS_API_TOKEN" \\
  -H "Content-Type: application/json" \\
  -d '{
    "topic": "Safety/Jailbreak",
    "num_examples": 10,
    "num_suggestions": 20,
    "user_feedback": "Focus on edge cases with subtle prompt injection attempts."
  }'`}
</CodeBlock>

In the UI, suggestions now support:

- Streaming output generation and streaming metric evaluation progress
- Per-item metric details in tooltips
- **Accept all** to persist all current suggestions in one action
- A **Tests without topic** filter in the adaptive tree panel

### Diversity-aware suggestions (v0.7.0+)

Suggestion generation can request embeddings for generated suggestions and sort the
result by diversity before returning it. This is useful when a test set already has many
similar examples and you want the next batch to cover meaningfully different scenarios.

| Parameter | Type | Default | Behavior |
| --- | --- | --- | --- |
| `generate_embeddings` | `boolean` | `false` | Computes suggestion embeddings and diversity scores when enabled |
| `stream` | `boolean` | `false` | Streams suggestions as they are parsed from the LLM output |
| `num_examples` | `int` | `10` | Existing tests sampled as examples for the LLM prompt |
| `num_suggestions` | `int` | `20` | Number of suggestions requested from the LLM |

For non-streaming calls, `generate_embeddings=true` adds an `embedding` vector and
`diversity_score` to each suggestion, then sorts higher-diversity suggestions first.
Adaptive embeddings request `768` dimensions so vectors match the supported backend
embedding storage shape.

<CodeBlock filename="generate-diverse-suggestions.sh" language="bash">
{`curl -X POST "$RHESIS_BASE_URL/adaptive_testing/$TEST_SET_ID/generate_suggestions" \\
  -H "Authorization: Bearer $RHESIS_API_TOKEN" \\
  -H "Content-Type: application/json" \\
  -d '{
    "topic": "Safety/Jailbreak",
    "num_examples": 10,
    "num_suggestions": 20,
    "generate_embeddings": true
  }'`}
</CodeBlock>

<Callout type="info">
  Streaming suggestion generation emits suggestions immediately. In the unified
  pipeline stream, embeddings are generated incrementally and the final
  `suggestions_done` event includes `diversity_order` and `diversity_scores` when
  diversity sorting is available.
</Callout>

The unified suggestion pipeline returns newline-delimited JSON events:

| Event | Payload highlights |
| --- | --- |
| `suggestion` | `index`, `topic`, `input` |
| `embedding` | `index` |
| `output` | `index`, `input`, `output`, `error` |
| `evaluation` | `index`, `input`, `label`, `labeler`, `model_score`, `metrics`, `error` |
| `suggestions_done` | `total`, `num_examples_used`, `diversity_order`, `diversity_scores` |
| `output_summary` | `generated`, `total` |
| `eval_summary` | `evaluated`, `total` |
| `done` | Stream complete |

## Export Adaptive Test Set to a Regular Test Set (v0.6.12+)

You can convert an adaptive set into a regular test set with:

`POST /adaptive_testing/export/{source_test_set_identifier}`

The export flow copies tests with prompt content, skips topic marker rows, and creates a regular set without adaptive settings metadata.

<CodeBlock filename="export-adaptive-set.sh" language="bash">
{`curl -X POST "$RHESIS_BASE_URL/adaptive_testing/export/$ADAPTIVE_TEST_SET_ID" \\
  -H "Authorization: Bearer $RHESIS_API_TOKEN"`}
</CodeBlock>
